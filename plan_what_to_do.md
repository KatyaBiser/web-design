### **Концепция проекта: "Rehab from Vibe"**

**Идея:** Создать сайт вымышленного реабилитационного центра "Rehab from Vibe". Этот центр помогает разработчикам, которые чрезмерно увлеклись "вайб-кодингом" — слепой зависимостью от ИИ-инструментов при написании кода. Сайт должен не только соответствовать всем техническим требованиям вашего университетского проекта, но и иронично обыгрывать проблему, предлагая "лечение" в виде возвращения к фундаментальным знаниям и осознанной разработке.

**Цель:** Продемонстрировать навыки качественной, продуманной веб-разработки (семантика, доступность, CSS-архитектура) как антитезу быстрому, но потенциально небезопасному и поверхностному коду, сгенерированному ИИ. В документации к проекту будет сделан особый акцент на том, почему были выбраны именно такие подходы, а не просто "потому что так сгенерировал ИИ".

### **3. Требования к HTML/CSS (интеграция с презентациями)**

Этот раздел — прямое руководство к действию для кодирования, основанное на ваших файлах.

**Общие требования:**
*   **Структура:** Создать **главную страницу + одну детальную** (например, профиль терапевта) в HTML/CSS. Для всех остальных страниц сделать кликабельный прототип.
*   **Подход:** Строго **Mobile-first**.
*   **Доступность (Accessibility):** Уровень **WCAG 2.0 AA**. Это означает:
    *   Весь интерактив должен быть доступен с клавиатуры.
    *   Контрастность текста и фона должна быть достаточной.
    *   У всех изображений, несущих смысл, должен быть `alt` атрибут. Декоративным — пустой `alt=""`.
    *   Использование ARIA-ролей для улучшения навигации (`role="navigation"`, `role="main"` и т.д.).

**HTML (на основе презентации "HTML")**
*   **Doctype:** `<!DOCTYPE html>`.
*   **Семантика:** Активно использовать семантические теги HTML5, как указано в требованиях и лекции:
    *   `<header>`: для шапки сайта.
    *   `<footer>`: для подвала.
    *   `<main>`: для основного контента страницы.
    *   `<nav>`: для основной навигации.
    *   `<article>`: для постов в блоге или карточек специалистов (как самодостаточных единиц контента).
    *   `<section>`: для группировки логических частей страницы (например, секция "О нас", секция "Программы").
    *   `<aside>`: для боковой панели, если она будет.
*   **Иерархия заголовков:** Соблюдать правильную иерархию `<h1>` - `<h6>`. `<h1>` должен быть один на странице.
*   **Формы:** Использовать `<form>`, `<input>`, `<label>`, `<textarea>`, `<select>`. Связывать `<label>` с `<input>` через `for` и `id`.
*   **Таблицы:** Использовать `<table>` для табличных данных (например, расписание курсов), а не для верстки. Обязательно использовать `<thead>`, `<tbody>`, `<caption>`.

**CSS (на основе презентации "Cascading Styles Sheets")**
*   **Без фреймворков:** Никаких Bootstrap, Foundation и т.д.
*   **Один файл CSS:** `style.css`, не менее 400 строк.
*   **CSS переменные:** Использовать для цветов, шрифтов, отступов для легкой смены темы.
    ```css
    :root {
      --primary-color: #3498db;
      --text-color: #333;
      --base-font-size: 16px;
    }
    ```
*   **Grid Layout:** Использовать для основной раскладки страницы (например, главный контент + боковая панель). Внутри грида должно быть не менее 8 элементов, 2 из которых занимают несколько ячеек.
*   **Flexbox:** Использовать для более мелких компонентов: выравнивание элементов в шапке, карточки в ряд и т.д.
*   **Адаптивность (Responsive Design):** Использовать **Media Queries** для адаптации под разные экраны, следуя принципу Mobile-first.
    ```css
    /* Стили для мобильных */
    .container { width: 100%; }

    /* Стили для планшетов и больше */
    @media (min-width: 768px) {
      .container { width: 90%; }
    }
    ```
*   **Методология:** В документации можно упомянуть, что для организации CSS использовались принципы **BEM (Block, Element, Modifier)** для именования классов, чтобы избежать хаоса и повысить переиспользуемость. Это будет отличным аргументом против хаотичного ИИ-кода.
    *   Пример: `.therapist-card__photo`, `.therapist-card__name--highlighted`.

### **4. Требования к кликабельному прототипу (Wireframe)**

*   **Инструмент:** Рекомендуется **Axure RP**, но можно использовать и другие.
*   **Что должен включать:**
    *   Прототипы **всех** страниц сайта (главная, списки, детальные страницы, контакты и т.д.).
    *   Версии для **мобильных устройств и десктопа**.
    *   **Интерактивность:** Все ссылки в меню, кнопки и карточки должны быть кликабельны и вести на соответствующие страницы прототипа.
*   **Формат сдачи:** Экспорт в HTML.

    *   **Ограничение ИИ:** Четко прописать, что ИИ использовался **только как ассистент** для поиска информации или рефакторинга мелких участков, а не для генерации всей архитектуры и логики. Привести примеры: "Я спросил у ИИ, как лучше реализовать `clamp()` для адаптивного шрифта, но всю структуру CSS Grid я спроектировал сам, исходя из макета".
*   **Методика проекта:**
    *   **Использование AI-инструментов:**
        *   **Какие были использованы:** Указать конкретные инструменты (например, ChatGPT, Copilot).
        *   **Как были использованы:** Привести примеры промптов и критически оценить полученные ответы. "ИИ предложил использовать `!important`, но я отказался от этого решения, так как оно нарушает специфичность CSS. Вместо этого я повысил специфичность селектора".
        *   **Критическая оценка:** Дать оценку точности и надежности ИИ. Указать на риски, такие как генерация устаревшего или небезопасного кода, и важность проверки со стороны разработчика.
